## Requirements
### Requirement: Применение плана через shadow-utils

`census apply` ДОЛЖЕН (MUST) материализовать план (Create/Update/Delete роль-учёток) через
системные утилиты shadow-utils (`useradd`/`usermod`/`groupadd`/`gpasswd`/`userdel`, `chfn`,
`passwd -l`) и НЕ ДОЛЖЕН (MUST NOT) редактировать `/etc/passwd`, `/etc/shadow`, `/etc/group`,
`/etc/gshadow` напрямую. Утилиты ДОЛЖНЫ (MUST) вызываться argv-массивом (без shell-строки),
успех/неуспех определяется кодом возврата. Создание учётки ДОЛЖНО (MUST) использовать явный
стабильный UID из декларации; смена UID существующей managed-учётки ДОЛЖНА (MUST) отвергаться
с диагностикой (UID стабилен по парку), а не выполняться перезаписью.

#### Scenario: Создание роль-учётки
- **WHEN** план содержит Create для роли с uid/shell/группами
- **THEN** учётка создаётся `useradd` с этим uid и shell, членство в группах выставляется, без shell-инъекций

#### Scenario: Конфликт UID
- **WHEN** план обновляет managed-учётку, у которой изменился бы UID
- **THEN** apply отвергает операцию с диагностикой, UID не перезаписывается

### Requirement: Атомарность и откат через full-file backup

Перед любой мутацией apply ДОЛЖЕН (MUST) снять снапшот целиком файлов `/etc/passwd`,
`/etc/shadow`, `/etc/group`, `/etc/gshadow` и каждого затрагиваемого `/etc/sudoers.d/census-*`
в каталог `0700 root`. При ошибке любой фазы apply ДОЛЖЕН (MUST) восстановить эти файлы из
снапшота (atomic rename) и завершиться ненулевым кодом, оставляя ОС в согласованном прежнем
состоянии (fail-closed). Реестр managed ДОЛЖЕН (MUST) обновляться атомарно последним, только
после успеха всех фаз.

#### Scenario: Сбой фазы apply
- **WHEN** мутация в середине apply завершается ошибкой
- **THEN** auth-базы восстанавливаются из снапшота, реестр не меняется, apply возвращает ошибку

#### Scenario: Успешный apply
- **WHEN** все фазы прошли успешно
- **THEN** реестр managed обновляется атомарно последним, состояние соответствует плану

### Requirement: Недостижимость роль-учётки при создании

Создаваемая роль-учётка ДОЛЖНА (MUST) немедленно получать заблокированный пароль (`passwd -l`,
поле shadow начинается с `!`) и НЕ ДОЛЖНА (MUST NOT) иметь `~/.ssh/authorized_keys`. Shell
роль-учётки — реальный (вход гейтит отсутствие учётных данных и cert-путь Tessera, не shell).
apply НЕ ДОЛЖЕН (MUST NOT) настраивать PAM-стек (зона Tessera Login) и НЕ ДОЛЖЕН (MUST NOT)
назначать parsec-метку-потолок (зона коммерческого ParsecBackend).

#### Scenario: Недостижимость созданной учётки
- **WHEN** роль-учётка создана apply
- **THEN** пароль заблокирован, authorized_keys отсутствует, su (non-root)/ssh/парольный вход отвергаются

### Requirement: Маркер managed — авторитет реестра

Census ДОЛЖЕН (MUST) трогать или удалять объект (учётку, группу, sudoers-файл), ТОЛЬКО если он
присутствует в root-only реестре `/var/lib/census/managed.toml`. Объект с GECOS-меткой Census,
но без записи в реестре, НЕ ДОЛЖЕН (MUST NOT) изменяться или удаляться (возможно чужой/
поддельный). GECOS-маркер ДОЛЖЕН (MUST) ставиться через `chfn` в формате без символов `:` и `=`
(ограничение `useradd -c` на Astra), и является лишь вторичным человекочитаемым признаком, не
авторитетом.

#### Scenario: Чужая учётка с поддельной GECOS-меткой
- **WHEN** на устройстве есть учётка с GECOS-меткой Census, но её нет в реестре managed
- **THEN** apply не трогает и не удаляет эту учётку

### Requirement: Anti-lockout gate

apply ДОЛЖЕН (MUST) отказываться выполнять план, если после применения не останется ни одного
рабочего пути входа на устройство. Rescue/break-glass-канал (определённый вне Census) НЕ ДОЛЖЕН
(MUST NOT) заноситься в реестр managed и тем самым структурно остаётся вне области изменений
Census. Если rescue-канал не сконфигурирован, apply ДОЛЖЕН (MUST) требовать явного флага
осознанного риска и логировать это решение, а не молча запирать устройство.

#### Scenario: План сносит последний путь входа
- **WHEN** план удалил бы последнюю рабочую точку входа, rescue отсутствует, флаг риска не задан
- **THEN** apply отвергает план до любых мутаций

#### Scenario: Rescue вне managed
- **WHEN** apply выполняется при наличии rescue-канала вне реестра managed
- **THEN** rescue-канал не входит в область изменений и не затрагивается

### Requirement: Валидация sudoers.d

Файл `sudoers.d` роли ДОЛЖЕН (MUST) записываться через временный файл с проверкой `visudo -c`
и атомарным rename; файл, не прошедший `visudo -c`, НЕ ДОЛЖЕН (MUST NOT) активироваться. Census
ДОЛЖЕН (MUST) владеть только файлами `census-<role>` и НЕ ДОЛЖЕН (MUST NOT) изменять чужие
sudoers-файлы.

#### Scenario: Битый sudoers
- **WHEN** сгенерированный sudoers-файл не проходит `visudo -c`
- **THEN** файл не активируется, фаза завершается ошибкой и откатывается

### Requirement: Доверие к декларации до мутаций (fail-closed)

apply ДОЛЖЕН (MUST) применять план, только если декларация доверена: подписана и прошла
anti-rollback (managed-режим), ЛИБО задан явный флаг `--trust-fs` (standalone, доверие
целостности ФС, логируется). При недоверенной или невалидной декларации apply ДОЛЖЕН (MUST)
отказывать до снятия снапшота и любых мутаций.

#### Scenario: Standalone-доверие
- **WHEN** apply запущен с `--trust-fs` на root-only `/etc/census/`
- **THEN** план применяется, решение о доверии ФС логируется

#### Scenario: Недоверенная декларация
- **WHEN** apply запущен без `--trust-fs` и без валидной подписи
- **THEN** apply отказывает до любых мутаций (fail-closed)

### Requirement: Идемпотентность apply

Повторный `census apply` той же декларации на согласованном устройстве ДОЛЖЕН (MUST) быть
no-op: пустой план → ноль мутаций. apply НЕ ДОЛЖЕН (MUST NOT) изменять объекты, уже
соответствующие декларации.

#### Scenario: Повторный apply
- **WHEN** apply запущен дважды подряд с той же декларацией
- **THEN** второй запуск не выполняет мутаций (план пуст)
