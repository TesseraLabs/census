## Requirements
### Requirement: Роль описывается через разрешения

Census ДОЛЖЕН (MUST) принимать в role-slice поле `payload.permissions` — список разрешений
(id-строка или таблица `{id, <параметры>}`) — и раскрывать его в Unix-примитивы через каталог.
Сырые `payload.groups`/`sudo_role`/`limits` ДОЛЖНЫ (MUST) оставаться допустимыми как escape
hatch и объединяться (union) с раскрытием разрешений; их использование помимо разрешений
ДОЛЖНО (MUST) помечаться lint-предупреждением.

#### Scenario: Роль в разрешениях раскрывается в примитивы
- **WHEN** role-slice несёт `permissions = ["network-admin", "log-read"]`
- **THEN** резолв даёт объединённые группы/sudo-команды/лимиты этих разрешений для ОС-цели устройства

#### Scenario: Сырые примитивы как escape hatch
- **WHEN** role-slice несёт и `permissions`, и сырой `groups`
- **THEN** итог — union раскрытия и сырых групп, с lint-предупреждением о сырых примитивах

### Requirement: Формат и строгий парсинг записи каталога

Запись каталога ДОЛЖНА (MUST) парситься строго (`deny_unknown_fields`): известные поля — `id`,
`risk` (`contained` | `escalation-capable`), `category`, раскрытие `groups`/`sudo`/`limits`,
агрегация `includes`/`include_categories`. Незнакомое поле ДОЛЖНО (MUST) быть ошибкой. Раскрытие
НЕ ДОЛЖНО (MUST NOT) включать `mac_mask` (МКЦ-потолок — отдельный коммерческий слой, не Census).

#### Scenario: Незнакомое поле в записи каталога
- **WHEN** запись политики содержит поле вне известной схемы
- **THEN** парсинг каталога отвергает запись с диагностикой

#### Scenario: Раскрытие только в учётко-слой
- **WHEN** разрешение раскрывается
- **THEN** результат содержит только groups/sudo/limits, никогда mac_mask

### Requirement: Слоёный резолв по ОС-цели

Census ДОЛЖЕН (MUST) определять ОС-цель из `/etc/os-release` (`ID`, `VERSION_ID`) и резолвить
каталог по цепочке слоёв снизу вверх (`linux` → `linux-<distro>` → `linux-<distro>-<ver>` →
overlay `/etc`), применяя точечные (`<field>.append`, переопределение поля) и полные (`replace`)
переопределения; provenance ДОЛЖЕН (MUST) фиксировать слой-источник каждого примитива. ОС-цель
ДОЛЖНА (MUST) переопределяться флагом/полем декларации. Незнакомая версия ДОЛЖНА (MUST)
резолвиться ближайшим известным слоем ниже с lint-предупреждением, НЕ молча как последняя.

#### Scenario: Версия-слой дополняет дистрибутив
- **WHEN** `linux-debian` даёт sudo `[ip]`, а `linux-debian-12` добавляет `sudo.append = [netplan]`
- **THEN** на debian-12 раскрытие = `[ip, netplan]`, provenance помечает слой каждого

#### Scenario: Незнакомая версия дистрибутива
- **WHEN** os-release сообщает версию без своего слоя в каталоге
- **THEN** резолв берёт ближайший известный нижний слой и выдаёт lint-предупреждение

### Requirement: Бандлы и категории

Запись каталога ДОЛЖНА (MUST) уметь агрегировать другие через `includes` (явные id) и
`include_categories` (все разрешения категории); резолв ДОЛЖЕН (MUST) разворачивать их
транзитивно, отвергать циклы как ошибку, и выводить риск бандла как максимум рисков членов
(явный `risk` ниже выведенного — ошибка). Членство по `include_categories` ДОЛЖНО (MUST)
материализоваться против зафиксированного `catalog_version` и записываться в provenance, чтобы
добавление разрешения в категорию позже НЕ расширяло уже скомпилированную роль молча.

#### Scenario: Бандл раскрывается в union членов
- **WHEN** роль ссылается на бандл `network-config` с `includes = ["network-diag", "network-admin"]`
- **THEN** раскрытие = union примитивов членов плюс собственные примитивы бандла

#### Scenario: Цикл во включениях
- **WHEN** бандл A включает B, а B включает A
- **THEN** компиляция отвергает каталог с ошибкой цикла

#### Scenario: Категория материализуется против версии каталога
- **WHEN** бандл задан `include_categories = ["network"]`
- **THEN** в provenance фиксируется конкретный список членов на текущий catalog_version

### Requirement: Add-on пакеты стороннего ПО через namespace

Census ДОЛЖЕН (MUST) обнаруживать разрешения в namespace-подкаталогах (`<os-target>/<namespace>/`)
наравне с базовыми, требовать namespace-префикс у id вне базы (`docker.ps`), и отвергать как
ошибку коллизию одного id из разных источников. Ссылка роли на разрешение отсутствующего add-on'а
ДОЛЖНА (MUST) давать ошибку резолва до apply (не молчаливый пропуск). Top-level (id без namespace)
ДОЛЖЕН (MUST) быть зарезервирован под примитивы ОС.

#### Scenario: Add-on присутствует
- **WHEN** установлен add-on `docker` и роль ссылается `permissions = ["docker.admin"]`
- **THEN** Census обнаруживает `docker.*` в namespace-подкаталоге и раскрывает его

#### Scenario: Add-on отсутствует
- **WHEN** роль ссылается `docker.admin`, но add-on `docker` не установлен
- **THEN** резолв завершается ошибкой до apply с указанием неизвестного разрешения

#### Scenario: Коллизия id
- **WHEN** два источника определяют один и тот же id
- **THEN** компиляция отвергает каталог с ошибкой коллизии

### Requirement: Локализация описаний отдельным деревом

Человеко-тексты (`title`, `summary`, `risk_note`) ДОЛЖНЫ (MUST) храниться отдельно от записей
политик — в дереве `l10n/<locale>/<group>.toml` с секциями по id, os-агностично; overlay
`/etc` ДОЛЖЕН (MUST) перекрывать `/usr/share` для той же локали+id. Отображение ДОЛЖНО (MUST)
делать fallback `запрошенная локаль → en → id`. Тексты — метаданные и НЕ ДОЛЖНЫ (MUST NOT)
влиять на раскрытие примитивов, классы риска, резолв или проверки прав; отсутствующий/битый
перевод НЕ ДОЛЖЕН (MUST NOT) ломать apply (lint предупреждает о пропусках и осиротевших ключах).

#### Scenario: Перевод подхватывается по локали
- **WHEN** есть `l10n/ru/network.toml` с `[network-admin] title=…` и запрошен `--lang ru`
- **THEN** `census show` рендерит русский текст для `network-admin`

#### Scenario: Fallback при отсутствии перевода
- **WHEN** для запрошенной локали нет текста разрешения, а для `en` есть
- **THEN** отображается английский; при отсутствии и его — сам id; apply не затронут

### Requirement: Compile встроен в резолв, provenance, неизменность доверия

Раскрытие разрешений ДОЛЖНО (MUST) выполняться в ходе резолва role-store, до построения плана,
так что `plan`/`apply` работают с ролями в разрешениях напрямую. Скомпилированный срез ДОЛЖЕН
(MUST) нести provenance: для каждого примитива — id-источник, бандл (если через бандл), слой
каталога и `catalog_version`. В managed-режиме подписи ДОЛЖЕН (MUST) подписываться уже
скомпилированный срез примитивов; верификация подписи НЕ ДОЛЖНА (MUST NOT) меняться каталогом.

#### Scenario: apply раскрывает разрешения до плана
- **WHEN** `census apply` получает роль с `permissions`
- **THEN** разрешения раскрываются в примитивы до diff, план строится по примитивам

#### Scenario: Provenance скомпилированного среза
- **WHEN** разрешение раскрылось в sudo-команду
- **THEN** срез фиксирует id-источник, слой каталога и catalog_version этой команды

### Requirement: Классы риска advisory

Класс риска разрешения (`contained` | `escalation-capable`) ДОЛЖЕН (MUST) показываться в
`census show`/`compile` и lint и НЕ ДОЛЖЕН (MUST NOT) сам по себе блокировать раскрытие или apply
(честная маркировка, не enforcement). Заведомо root-эквивалентные способности ДОЛЖНЫ (MUST)
маркироваться `escalation-capable` без занижения.

#### Scenario: Эскалируемое разрешение помечено честно
- **WHEN** разрешение даёт путь до root (например членство в группе docker)
- **THEN** оно отображается как escalation-capable, apply не блокируется этим фактом

### Requirement: CLI compile и show

Census ДОЛЖЕН (MUST) предоставлять `census compile <role>` (плоский скомпилированный срез +
provenance; опции `--os-target`, `--catalog-dir`, `--lint`) и `census show <role> --lang <l>`
(дерево разрешение/бандл → примитив с локализованными текстами и классами риска). Обе команды
read-only и НЕ ДОЛЖНЫ (MUST NOT) мутировать систему.

#### Scenario: compile печатает срез и provenance
- **WHEN** запущен `census compile oper`
- **THEN** печатается плоское раскрытие примитивов с источником каждого, без мутаций

#### Scenario: show рендерит дерево с локализацией
- **WHEN** запущен `census show oper --lang ru`
- **THEN** печатается дерево разрешений→примитивов с русскими описаниями и классами риска

### Requirement: Lint каталога и роли

Census ДОЛЖЕН (MUST) предоставлять lint (`census compile --lint` и/или часть `doctor`),
выявляющий: сырые примитивы помимо разрешений, незнакомую ос-цель/версию, переопределение
vendor-разрешения слоем заказчика без явного флага, изменение членства бандла между версиями
каталога, занижение риска бандла, недостающие и осиротевшие переводы, коллизию namespace, и id
без namespace от источника вне базы. Lint ДОЛЖЕН (MUST) возвращать ненулевой код при ошибках
(для CI), отделяя их от предупреждений.

#### Scenario: lint ловит переопределение vendor без флага
- **WHEN** слой `/etc` переопределяет vendor-разрешение без явного флага override
- **THEN** lint выдаёт предупреждение об этом переопределении

#### Scenario: lint ошибка даёт ненулевой код
- **WHEN** в каталоге есть коллизия namespace
- **THEN** `census compile --lint` завершается ненулевым кодом
