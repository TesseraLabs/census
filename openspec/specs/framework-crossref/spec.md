## Requirements
### Requirement: Манифест фреймворка декларирует форму

Каждый фреймворк ДОЛЖЕН (MUST) иметь файл `framework.toml` с полями `id`, `version`, `title`,
`dimension` (`flat` | `os-layered`) и `provides` (список тегов провайдеров, напр. `crossref`,
`controls`). Загрузчик ДОЛЖЕН (MUST) применять форму резолва согласно `dimension`. Незнакомый
`dimension` или незнакомый тег в `provides` ДОЛЖЕН (MUST) приводить к пропуску фреймворка с
предупреждением (forward-compat: данные новее бинаря), НЕ к ошибке.

#### Scenario: flat-фреймворк
- **WHEN** `framework.toml` объявляет `dimension = "flat"`
- **THEN** загрузчик читает `mappings/*.toml` напрямую, без ос-цепочки

#### Scenario: os-layered фреймворк
- **WHEN** `framework.toml` объявляет `dimension = "os-layered"`
- **THEN** загрузчик резолвит `mappings/<os>/*.toml` через цепочку слоёв каталога

#### Scenario: Незнакомый dimension
- **WHEN** манифест объявляет неизвестный `dimension`
- **THEN** фреймворк пропускается с предупреждением, остальные грузятся, код не валится

### Requirement: Формат файла маппинга с полярностью связи

Файл маппинга ДОЛЖЕН (MUST) быть TOML, где ключ верхнего уровня = id разрешения каталога.
Связь разрешения с контролем ДОЛЖНА (MUST) различать полярность через три необязательных
списка control-id: `satisfies` (наличие способности адресует контроль), `risk` (способность
контроль подрывает), `related` (касается области контроля нейтрально). Все три по умолчанию
пусты; разрешение может иметь любую комбинацию. Один и тот же control-id в `satisfies` и `risk`
одного разрешения ДОЛЖЕН (MUST) давать ошибку lint (противоречие). Все файлы `*.toml` одного
фреймворка ДОЛЖНЫ (MUST) мёржиться по имени; при дубле permission-id списки каждой полярности
ДОЛЖНЫ (MUST) объединяться с устранением дублей. Парсинг ДОЛЖЕН (MUST) быть строгим к структуре
(deny_unknown_fields) и толерантным к незнакомым permission-id (forward-compat).

#### Scenario: Связь satisfies
- **WHEN** файл содержит `[sudo-config]` с `satisfies = ["7.2.2", "7.3.1"]`
- **THEN** загрузчик связывает `sudo-config` с `7.2.2` и `7.3.1` как удовлетворяющие

#### Scenario: Связь risk
- **WHEN** файл содержит `[log-admin]` с `risk = ["10.5.1"]`
- **THEN** загрузчик связывает `log-admin` с `10.5.1` как угрозу, НЕ как покрытие

#### Scenario: Противоречие satisfies/risk
- **WHEN** один control-id указан и в `satisfies`, и в `risk` одного разрешения
- **THEN** lint выдаёт ошибку

#### Scenario: Аддитивный мёрж дублей
- **WHEN** два файла одного фреймворка задают связи для одного permission-id
- **THEN** списки каждой полярности = объединение обоих без дублей

### Requirement: Перечень контролей и граница домена

Если `provides` фреймворка включает `controls`, фреймворк ДОЛЖЕН (MUST) поставлять
`controls.toml`, где каждый контроль имеет `title` и обязательное булево поле `owned`.
Отсутствие `owned` у контроля ДОЛЖНО (MUST) давать ошибку lint. Контроли с `owned = false`
ДОЛЖНЫ (MUST) трактоваться как «вне домена Census» и НЕ ДОЛЖНЫ (MUST NOT) учитываться в расчёте
покрытия.

#### Scenario: Контроль без owned
- **WHEN** запись `controls.toml` не содержит поля `owned`
- **THEN** lint выдаёт ошибку

#### Scenario: Контроль вне домена
- **WHEN** контроль помечен `owned = false`
- **THEN** gap-расчёт его игнорирует, а вывод помечает как вне домена Census

### Requirement: Слой read-only, развязан от выдачи прав

Framework-слой ДОЛЖЕН (MUST) быть строго read-only: он НЕ ДОЛЖЕН (MUST NOT) влиять на `compile`,
раскрытие примитивов, `plan`, `apply` или реестр managed. Отсутствие дерева `frameworks/`
ДОЛЖНО (MUST) оставлять поведение Census неизменным. Битый или невалидный framework-файл
ДОЛЖЕН (MUST) давать предупреждение и НЕ ДОЛЖЕН (MUST NOT) прерывать `apply`.

#### Scenario: Нет дерева фреймворков
- **WHEN** каталог `frameworks/` отсутствует
- **THEN** `plan`/`apply`/`compile` работают как прежде, framework-команды сообщают об отсутствии данных

#### Scenario: Битый файл не ломает apply
- **WHEN** один framework-файл невалиден
- **THEN** выдаётся предупреждение, `apply` выполняется штатно

### Requirement: Инспекция роли по фреймворку

`census show <role> --framework <fw|all>` ДОЛЖЕН (MUST) для каждого разрешения роли печатать
связанные control-id указанного фреймворка (или всех при `all`) с provenance и **полярностью
связи** (satisfies / risk / related — визуально различимо). Разрешение без маппинга ДОЛЖНО
(MUST) явно помечаться как непокрытое, НЕ опускаться. Флаг `--format json` ДОЛЖЕН (MUST)
выдавать машиночитаемый вывод с полярностью каждой связи.

#### Scenario: Роль с satisfies-связью
- **WHEN** оператор запускает `census show serv --framework pci-dss`
- **THEN** рядом с каждым разрешением печатаются control-id, помеченные satisfies, и источник

#### Scenario: Роль с risk-связью
- **WHEN** разрешение роли имеет risk-связь с контролем
- **THEN** контроль печатается с пометкой угрозы (не как удовлетворённый)

#### Scenario: Разрешение без маппинга
- **WHEN** у разрешения роли нет маппинга в выбранном фреймворке
- **THEN** оно печатается с явной пометкой «нет маппинга»

### Requirement: Подкоманда framework и gap-oracle

`census framework` ДОЛЖЕН (MUST) предоставлять: `list` (установленные фреймворки с `version` и
`provides`), `show <fw>` (перечень контролей и статистика покрытия), `coverage <fw>` (gap-oracle),
`risk <fw>` (контроли под угрозой). `coverage` ДОЛЖЕН (MUST) вычислять непокрытые контроли как
множество `owned = true` контролей минус множество контролей, имеющих хотя бы одну
**`satisfies`-связь** (risk/related НЕ покрывают). `risk <fw>` ДОЛЖЕН (MUST) перечислять контроли,
имеющие хотя бы одну `risk`-связь, и угрожающие им разрешения, **независимо от `owned`** (угроза
контролю вне домена тоже значима). Все подкоманды ДОЛЖНЫ (MUST) поддерживать `--format json`.

#### Scenario: Список фреймворков
- **WHEN** оператор запускает `census framework list`
- **THEN** печатаются установленные фреймворки с их версией и провайдерами

#### Scenario: Непокрытые контроли
- **WHEN** оператор запускает `census framework coverage pci-dss`
- **THEN** печатаются покрытые (по satisfies) и непокрытые `owned = true` контроли; вне-доменные помечены отдельно

#### Scenario: Контроли под угрозой
- **WHEN** оператор запускает `census framework risk pci-dss`
- **THEN** печатаются контроли с risk-связью и разрешения-угрозы, включая контроли с `owned = false`

### Requirement: Валидация и lint framework-слоя

Lint ДОЛЖЕН (MUST) выдавать предупреждение при: осиротевшем маппинге (permission-id нет в
каталоге); рассинхроне `provides` и поставленных файлов; ссылке маппинга на control-id вне
`controls.toml`; дельте членства `controls.toml` между версиями фреймворка. Проверки
осиротевшего маппинга и неизвестного control-id ДОЛЖНЫ (MUST) охватывать все три полярности
(`satisfies`/`risk`/`related`). Один control-id в `satisfies` и `risk` одного разрешения
ДОЛЖЕН (MUST) давать ошибку (противоречие). Коллизия `framework.id` между двумя каталогами
ДОЛЖНА (MUST) давать ошибку (детерминизм).

#### Scenario: Осиротевший маппинг
- **WHEN** маппинг ссылается на permission-id, отсутствующий в каталоге
- **THEN** lint выдаёт предупреждение

#### Scenario: Коллизия id фреймворка
- **WHEN** два каталога объявляют один `framework.id`
- **THEN** lint выдаёт ошибку

### Requirement: Доверие — advisory с верификацией версии в отчёте

Vendor framework-данные ДОЛЖНЫ (MUST) доставляться подписанным пакетом; в standalone доверие =
права ФС. Поскольку слой не участвует в выдаче прав, подделка маппинга НЕ ДОЛЖНА (MUST NOT) вести
к эскалации привилегий — её максимальный эффект ограничен искажением compliance-аннотации.
Машиночитаемый вывод (`--format json`) ДОЛЖЕН (MUST) содержать `framework.id` и `version`,
служащие штампом версии для воспроизводимости.

#### Scenario: Штамп версии в выводе
- **WHEN** оператор запрашивает `--format json` у framework-команды
- **THEN** вывод содержит `id` и `version` фреймворка

#### Scenario: Подделка не эскалирует
- **WHEN** framework-файл подделан
- **THEN** искажается только compliance-аннотация; выдача прав и `apply` не затронуты
