## Requirements
### Requirement: Перечисление привилегированной поверхности

`census catalog coverage` ДОЛЖЕН (MUST) перечислять привилегированную поверхность устройства по
классам — setuid/setgid-бинари, sudo-достижимые бинари (`/usr/sbin`, `/sbin`, админ-`/usr/bin`),
security-relevant конфиги `/etc` (+ drop-in, conffiles), systemd-юниты, привратные к устройствам
группы, capability-несущие файлы — read-only, без запуска перечисляемых бинарей. Перечисление
ДОЛЖНО (MUST) пропускать виртуальные ФС и не пересекать сетевые точки монтирования.

#### Scenario: Перечисление классов поверхности
- **WHEN** запущен `census catalog coverage` на устройстве
- **THEN** собираются объекты классов sudo_bin/config/unit/group/capfile/setuid с провенансом (vendor/addon/orphan)

#### Scenario: Read-only без побочных эффектов
- **WHEN** идёт перечисление
- **THEN** перечисляемые бинари не запускаются, система не мутируется, содержимое конфигов не читается

### Requirement: Расчёт покрытия против раскрытого каталога

Покрытие ДОЛЖНО (MUST) считаться против РАСКРЫТЫХ примитивов установленного каталога (vendor +
add-on'ы + site-слой `/etc` + опционально роли) для ОС-цели устройства: `sudo_bin` покрыт при
префикс-матче пути sudo-строкой раскрытия на argv-границе; `config` — при path-glob-матче
`config-edit`-примитива; `unit` — при `service-admin` или совпадении имени с `service-restart`-
параметром (обе формы `<unit>`/`<unit>.service`); `group` — при вхождении в объединение
`groups`-раскрытий; `capfile` — при наличии `capability-admin`. Транзитивные побеги НЕ ДОЛЖНЫ
(MUST NOT) порождать фиктивное покрытие.

#### Scenario: Бинарь покрыт раскрытием sudo
- **WHEN** каталог раскрывает разрешение в sudo `/usr/sbin/ip` и на устройстве есть бинарь `/usr/sbin/ip`
- **THEN** этот sudo_bin помечается покрытым

#### Scenario: Непокрытый админ-бинарь
- **WHEN** на устройстве есть `/usr/sbin/cryptsetup`, не покрытый ни одним разрешением
- **THEN** он попадает в непокрытые с предложенным разрешением (suggested)

#### Scenario: Транзитивный побег не засчитывается
- **WHEN** бинарь достижим лишь через уже выданный shell
- **THEN** он не помечается покрытым по факту побега (покрыт только сам выданный бинарь)

### Requirement: Отчёт и CI-gate

`census catalog coverage` ДОЛЖЕН (MUST) выдавать человекочитаемый отчёт (сводка покрытия по
классам и общий %, непокрытые объекты по доменам с предложенным разрешением, секции
intentionally-uncovered с причиной и anomalies) и машиночитаемый `--json`. Флаг
`--min-coverage <pct>` ДОЛЖЕН (MUST) приводить к ненулевому коду возврата при общем покрытии ниже
порога (CI-gate). Намеренно непокрытые объекты (механизмы эскалации, МКЦ/pdpl, app/admin-группы,
шумовые группы) НЕ ДОЛЖНЫ (MUST NOT) штрафовать метрику и ДОЛЖНЫ (MUST) помечаться причиной.

#### Scenario: CI-gate по порогу покрытия
- **WHEN** `census catalog coverage --min-coverage 85` и общее покрытие 81%
- **THEN** процесс завершается ненулевым кодом

#### Scenario: Намеренно непокрытое помечено причиной
- **WHEN** на устройстве есть `/usr/bin/pdpl-user` (МКЦ, коммерческий слой)
- **THEN** он показан в intentionally-uncovered с причиной, не как пробел, и не штрафует метрику

#### Scenario: Аномальный orphan-setuid
- **WHEN** найден setuid-бинарь вне владения пакетами (orphan)
- **THEN** он попадает в секцию anomalies (расследовать), отдельно от непокрытых vendor-объектов

### Requirement: ОС-цель и кросс-аудит

Покрытие ДОЛЖНО (MUST) браться из `/etc/os-release` (как apply) и сравниваться с тем же слоем
каталога, что применяется на устройстве; флаг `--os-target` ДОЛЖЕН (MUST) переопределять цель для
кросс-аудита (что было бы непокрыто на другом дистрибутиве).

#### Scenario: Покрытие для целевого дистрибутива
- **WHEN** устройство — astra и каталог имеет слой `linux-astra`
- **THEN** покрытие считается против astra-раскрытия (с его специфичными группами/утилитами)
