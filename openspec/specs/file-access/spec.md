## Requirements
### Requirement: Объявление файлового доступа в каталоге

Запись каталога ДОЛЖНА (MUST) уметь объявлять файловые гранты `[[file]]` с полями `path`
(абсолютный), `access` (`ro`|`rw`) и `recursive`. Путь ДОЛЖЕН (MUST) строго валидироваться
(абсолютный, без управляющих символов и компонента `..`); параметр `{name}` ДОЛЖЕН (MUST)
подставляться из параметров разрешения с пост-подстановочной валидацией результата (как у
параметризованных sudo-команд). Парсинг под-таблицы гранта ДОЛЖЕН (MUST) быть строгим
(`deny_unknown_fields`).

#### Scenario: Папочный грант объявлен
- **WHEN** разрешение несёт `[[file]] path="/etc/ssh" access="rw" recursive=true`
- **THEN** резолв даёт файловый грант на эту папку с доступом rw и provenance

#### Scenario: Невалидный путь отвергается
- **WHEN** грант содержит относительный путь или `..` или управляющий символ
- **THEN** парсинг/резолв отвергает запись с диагностикой

### Requirement: Сменный бэкенд enforcement'а с декларацией возможностей

Материализация файловых грантов ДОЛЖНА (MUST) идти через бэкенд, реализующий SPI
`FileAccessBackend`, который ДОЛЖЕН (MUST) декларировать свои возможности (dir/per_path/pattern/
realtime/rewrite_proof). Резолвер ДОЛЖЕН (MUST) подбирать для каждого гранта бэкенд, чьи
возможности покрывают форму пути гранта (папка/файл/шаблон). Census ДОЛЖЕН (MUST) ставить и
снимать ТОЛЬКО собственную запись доступа на роль-учётку и НЕ ДОЛЖЕН (MUST NOT) трогать владельца,
режим или чужие записи доступа.

#### Scenario: Папочный грант материализуется открытым бэкендом
- **WHEN** установлен `AclBackend` и грант — папочный rw
- **THEN** доступ роль-учётки выставляется на папку (с наследованием новыми файлами), чужие записи не затрагиваются

#### Scenario: Отзыв снимает только свою запись
- **WHEN** грант исчез из раскрытия и отзывается
- **THEN** снимается только запись роль-учётки; чужие записи доступа и владелец/режим целы

### Requirement: Папочный грант переживает перезапись файлов

Открытый `AclBackend` ДОЛЖЕН (MUST) применять папочный грант так, чтобы файлы, создаваемые в папке
(в т.ч. при правке через пересоздание и при ротации), наследовали доступ (default-ACL). Грант на
папку НЕ ДОЛЖЕН (MUST NOT) терять силу из-за перезаписи файла внутри неё.

#### Scenario: Новый файл в папке наследует доступ
- **WHEN** применён папочный rw-грант и в папке создаётся новый файл (правкой через rename или ротацией)
- **THEN** новый файл доступен роль-учётке без повторного apply

### Requirement: Capability-gating и честный отказ

apply и `compile --lint` ДОЛЖНЫ (MUST) отвергнуть до любой мутации (fail-closed) грант формы
файла или шаблона, если ни один установленный бэкенд не декларирует соответствующую возможность
(`per_path`/`pattern`), с диагностикой, предлагающей расширить грант до папки или установить
способный бэкенд. Census НЕ ДОЛЖЕН (MUST NOT) применять частичный или ненадёжный доступ вместо
запрошенного.

#### Scenario: Файловый грант без способного бэкенда
- **WHEN** объявлен грант на конкретный файл, а установлен только папочный `AclBackend`
- **THEN** apply отвергает до мутаций с подсказкой (расширить до папки или установить бэкенд)

#### Scenario: Шаблонный грант без способного бэкенда
- **WHEN** объявлен грант на шаблон `*.log`, а способного бэкенда нет
- **THEN** `compile --lint` завершается ненулевым кодом, грант не применяется

### Requirement: Откат файловых грантов

Перед мутацией доступа apply ДОЛЖЕН (MUST) снять снапшот затрагиваемых путей (через бэкенд) и
зарегистрировать их в backup-наборе; при ошибке фазы apply ДОЛЖЕН (MUST) восстановить прежний
доступ из снапшота (fail-closed), оставляя файловые права в согласованном прежнем состоянии.

#### Scenario: Сбой фазы откатывает доступ
- **WHEN** мутация файлового доступа в середине apply завершается ошибкой
- **THEN** прежний доступ восстанавливается из снапшота, apply возвращает ошибку

### Requirement: Учёт файловых грантов в покрытии

`census catalog coverage` ДОЛЖЕН (MUST) считать объект класса `config` покрытым, если его путь
совпадает с файловым грантом или находится под рекурсивным папочным грантом, различая ro/rw, и
ДОЛЖЕН (MUST) указывать в отчёте бэкенд и уровень гарантии. Знаменатель класса `config` ДОЛЖЕН
(MUST) ограничиваться security-relevant путями, а не всеми conffiles.

#### Scenario: Конфиг под папочным грантом покрыт
- **WHEN** каталог даёт rw-грант на `/etc/ssh`, а на устройстве есть `/etc/ssh/sshd_config`
- **THEN** этот config-объект помечается покрытым (с указанием бэкенда/гарантии)
