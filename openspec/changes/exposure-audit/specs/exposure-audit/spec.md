# exposure-audit Specification

## Purpose

Read-only аудит фактического состояния прав файловой системы. Отвечает на обратный к provisioning
вопрос: что принципал РЕАЛЬНО достижимо читает/пишет — сверх least-privilege замысла, — и какие
opasные классы прав (world-writable, setuid, world-readable секреты, broad-group-writable) лежат в
системе. Один permission-индекс, два режима: глобальная posture-карта (`audit fs`) и экспозиция под
принципал (`audit expose`). Слой ничего не мутирует; находки классифицируются по риску и пути
ремедиации (ambient / in-model).

## ADDED Requirements

### Requirement: Read-only инвариант и код возврата

Слой `exposure-audit` ДОЛЖЕН (MUST) выполнять только чтение и НЕ ДОЛЖЕН (MUST NOT) мутировать ни
один объект ОС, файловую систему или реестр managed. При находках уровня выше настраиваемого порога
severity процесс ДОЛЖЕН (MUST) завершаться ненулевым кодом; при чистом результате или только
находках ниже порога — нулевым. Любая ошибка не должна приводить к частичной мутации, так как
мутаций нет вовсе.

#### Scenario: Чистая система
- **WHEN** ни одна находка не достигает порога severity
- **THEN** аудит печатает отсутствие значимых находок и возвращает 0, ничего не изменив

#### Scenario: Находка выше порога
- **WHEN** аудит находит хотя бы одну находку уровня выше порога (напр. escalation на cron)
- **THEN** находка печатается, процесс возвращает ненулевой код, ОС не изменена

### Requirement: Permission-индекс и охват скана

Аудит ДОЛЖЕН (MUST) строить permission-индекс одним обходом охваченных корней, записывая на каждый
инод путь, владельца (uid), группу (gid), режим, POSIX ACL и класс объекта; индекс ДОЛЖЕН (MUST)
переиспользоваться всеми запросами одного прогона. По умолчанию охват ДОЛЖЕН (MUST) ограничиваться
настраиваемым набором security-relevant корней, а псевдо-файловые системы (`/proc`, `/sys`,
`/dev`) и сетевые монтирования ДОЛЖНЫ (MUST) пропускаться. Флаг `--root PATH` (повторяемый) ДОЛЖЕН
(MUST) задавать явные корни, `--full` — обход от `/` (кроме псевдо-ФС).

#### Scenario: Дефолтный охват
- **WHEN** аудит запущен без scope-флагов
- **THEN** обходятся только настроенные security-relevant корни, псевдо-ФС и сетевые ФС пропущены

#### Scenario: Явные корни
- **WHEN** заданы `--root /opt/app --root /srv`
- **THEN** индекс строится только по этим деревьям

### Requirement: Reachability с x-траверсией предков

Экспозиция ДОЛЖНА (MUST) считать объект достижимым для принципала только если КАЖДЫЙ предок-каталог
по пути от корня обхода до объекта даёт этому принципалу право поиска (`x`) по тому же access-check
алгоритму; если хотя бы один предок не даёт `x`, объект и его поддерево ДОЛЖНЫ (MUST) считаться
недостижимыми и не давать находку. Результат траверсии предков ДОЛЖЕН (MUST) мемоизироваться по
дереву каталогов. uid 0 ДОЛЖЕН (MUST) короткозамыкаться как достижимый.

#### Scenario: Файл за закрытым каталогом
- **WHEN** файл имеет режим 777, но один из предков — каталог 700, владелец root, а принципал не root и не владелец
- **THEN** объект считается недостижимым и НЕ даёт находку

#### Scenario: Достижимый world-writable
- **WHEN** все предки дают принципалу `x`, а целевой каталог world-writable
- **THEN** объект даёт находку записи (write) с указанием пути доступа

### Requirement: Резолв принципала и членство в группах

Аудит ДОЛЖЕН (MUST) принимать принципал как имя или числовой uid и резолвить его в uid, первичную и
дополнительные группы по семантике getgrouplist. Источником в этой версии ДОЛЖНЫ (MUST) быть
локальные `/etc/passwd` и `/etc/group`; неучёт NSS/LDAP-источников ДОЛЖЕН (MUST) быть
задокументирован как advisory-ограничение. Access-check для группового класса ДОЛЖЕН (MUST)
учитывать ВСЕ дополнительные группы принципала, не только первичную.

#### Scenario: Доступ через дополнительную группу
- **WHEN** объект group-writable для группы `app`, а принципал состоит в `app` как в дополнительной
- **THEN** объект даёт находку записи через `via=group:app`

#### Scenario: Принципал по числовому uid
- **WHEN** принципал задан как `--principal 5012`
- **THEN** резолв даёт uid 5012 и его группы из локальных файлов

### Requirement: ACL в access-check

Access-check ДОЛЖЕН (MUST) учитывать POSIX ACL: запись named-user на uid принципала применяется
с маской; при отсутствии named-user учитываются записи owning-group и named-group из числа групп
принципала, объединённые и маскированные ACL-маской; иначе применяются биты other. Чтение ACL
ДОЛЖНО (MUST) переиспользовать существующий механизм чтения POSIX ACL слоя file-access.

#### Scenario: Доступ через named-user ACL
- **WHEN** на объекте есть ACL `user:role-x:rw-` и маска разрешает `rw`
- **THEN** принципал `role-x` получает write через `via=acl_user:role-x`

#### Scenario: Маска срезает ACL-запись
- **WHEN** ACL даёт named-group `rw`, но ACL-маска = `r--`
- **THEN** эффективный доступ группы — только read

### Requirement: Глобальная posture-карта классов

`census audit fs` ДОЛЖЕН (MUST) перечислять principal-independent опасные классы из индекса:
world-writable объекты в чувствительных деревьях; инвентарь setuid/setgid бинарей с флагом тех, что
ещё и записываемы; world-readable объекты класса secret; объекты, записываемые широкими группами
(`users`, `staff` и настраиваемый список). Режим `audit fs` НЕ ДОЛЖЕН (MUST NOT) требовать
указания принципала.

#### Scenario: World-writable в чувствительном дереве
- **WHEN** в охвате есть каталог `/var/spool/cron` с режимом 777
- **THEN** он попадает в posture-карту как world-writable класса cron

#### Scenario: Записываемый setuid-бинарь
- **WHEN** есть setuid-бинарь, который вдобавок world-writable
- **THEN** он перечислен в инвентаре setuid с флагом writable (повышенная severity)

### Requirement: Taxonomy и деривация severity находки

Каждая находка ДОЛЖНА (MUST) нести поля: `principal` (null в режиме `audit fs`), `path`, `access`
(эффективные r/w/x), `via` (`other_bits`|`group:<g>`|`acl_user:<u>`|`acl_group:<g>`|`owner`),
`class` (`cron`|`systemd-unit`|`path-binary`|`sudoers`|`config`|`secret`|`setuid-binary`|
`generic`), `risk` (`escalation`|`leak`|`tamper`), `severity`, `remediation_class`, `hint`.
Severity ДОЛЖНА (MUST) выводиться детерминированно из класса и риска: write-эскалация на
cron/sudoers/systemd-unit/path-binary — high; read-leak на класс secret — high; world-writable в
generic/`/tmp`-подобном — low.

#### Scenario: Эскалация на cron = high
- **WHEN** принципал может писать в объект класса cron
- **THEN** находка имеет `risk=escalation` и `severity=high`

#### Scenario: Утечка секрета = high
- **WHEN** принципал может читать объект класса secret через other-биты
- **THEN** находка имеет `risk=leak`, `severity=high`, `via=other_bits`

### Requirement: Классификация пути ремедиации

Каждая находка ДОЛЖНА (MUST) нести `remediation_class`, честно различающий источник доступа: `ambient`
— источник есть чужой объект (режим/владелец/ACL не под управлением Census); `in-model` — источник
есть объект, которым владеет Census (раздутое членство в управляемой группе или собственный
file-access грант шире необходимого). Для `ambient` `hint` ДОЛЖЕН (MUST) предлагать ручную правку
(`chmod`/`setfacl`) и аудит НЕ ДОЛЖЕН (MUST NOT) утверждать, что Census это исправит. Для `in-model`
`hint` ДОЛЖЕН (MUST) указывать на сужение декларации. Генерация TOML-патча НЕ требуется в этой
версии (только классификация).

#### Scenario: Ambient world-writable
- **WHEN** доступ даёт world-writable чужой каталог
- **THEN** `remediation_class=ambient`, hint предлагает `chmod o-w <path>`, без обещания авто-фикса

#### Scenario: In-model раздутая группа
- **WHEN** доступ даёт членство в управляемой Census группе, которое шире нужного
- **THEN** `remediation_class=in-model`, hint предлагает сузить декларацию роли/группы

### Requirement: Вычитание intended baseline для managed-принципала

В режиме `audit expose` для принципала, являющегося Census-managed роль-учёткой, аудит ДОЛЖЕН (MUST)
вычитать intended baseline — её домашний каталог и пути, выданные ей через каталог разрешений — и
показывать только ЛИШНИЙ доступ сверх замысла. Для принципала, не являющегося managed (произвольный
uid), baseline отсутствует, и аудит ДОЛЖЕН (MUST) показывать сырой reachability без вычитания.

#### Scenario: Managed-учётка — только лишнее
- **WHEN** managed роль-учётке выдан грант на `/etc/ssh`, и она достижимо пишет туда И в world-writable `/var/spool/cron`
- **THEN** в экспозиции остаётся только `/var/spool/cron` (грант `/etc/ssh` вычтен как intended)

#### Scenario: Немэнэджед uid — сырой reachability
- **WHEN** принципал — произвольный uid вне реестра managed
- **THEN** показывается весь достижимый доступ без вычитания baseline

### Requirement: Честность DAC-only вердикта

Аудит ДОЛЖЕН (MUST) оценивать только дискреционный контроль доступа (DAC: режим, владелец, группы,
POSIX ACL) и НЕ ДОЛЖЕН (MUST NOT) учитывать MAC-слои (SELinux, AppArmor, PARSEC-метку). Вывод ДОЛЖЕН
(MUST) документировать, что вердикт является верхней границей доступа: MAC может дополнительно
ограничить фактический доступ.

#### Scenario: Пометка верхней границы
- **WHEN** аудит печатает находки экспозиции
- **THEN** в выводе присутствует пометка, что вердикт — DAC-only верхняя граница (MAC не учтён)

### Requirement: Вывод, стабильный контракт и неутечка секретов

Аудит ДОЛЖЕН (MUST) поддерживать `--format text|json`; JSON ДОЛЖЕН (MUST) быть стабильным
контрактом, зафиксированным golden-тестами (schemars/clap, как у прочих команд). Вывод НЕ ДОЛЖЕН
(MUST NOT) включать СОДЕРЖИМОЕ файлов класса secret — только путь, режим и метаданные находки, так
как сам отчёт является чувствительным артефактом (карта слабых мест).

#### Scenario: JSON-контракт
- **WHEN** аудит запущен с `--format json`
- **THEN** вывод соответствует зафиксированной golden-схеме находок

#### Scenario: Секрет не раскрывается
- **WHEN** находка касается объекта класса secret
- **THEN** в выводе есть путь и режим, но НЕ содержимое файла

### Requirement: Интерактивный выбор охвата

При запуске на интерактивном терминале (TTY) без явных scope-флагов аудит ДОЛЖЕН (MUST) предлагать
выбор охвата: только security-relevant корни (по умолчанию), полный обход от `/`, либо
пользовательский набор корней. В неинтерактивном окружении (нет TTY) аудит НЕ ДОЛЖЕН (MUST NOT)
блокироваться на запрос и ДОЛЖЕН (MUST) применять дефолтный охват.

#### Scenario: Интерактивный запуск
- **WHEN** аудит запущен на TTY без `--root`/`--full`
- **THEN** оператору предлагается выбор важное/полный/свои корни

#### Scenario: Неинтерактивный запуск
- **WHEN** аудит запущен без TTY и без scope-флагов
- **THEN** применяется дефолтный security-relevant охват без запроса
