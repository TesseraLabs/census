# live-reconcile Specification

## Purpose

Согласование плана `census apply` с реестром живых сессий Tessera: деструктив над роль-учёткой
с активной сессией откладывается, живая сессия не рвётся, владение учёткой удерживается до
безопасного удаления. Census не убивает сессии — только пропускает свой деструктив (§12).

## ADDED Requirements

### Requirement: Отсрочка удаления при живой сессии

`census apply` ДОЛЖЕН (MUST) перед фазой удаления прочитать реестр живых сессий Tessera и для
каждой роль-учётки с активной сессией **отложить** её удаление: `userdel` НЕ ДОЛЖЕН (MUST NOT)
выполняться, учётка исключается из исполняемого плана. Census НЕ ДОЛЖЕН (MUST NOT) завершать
живые сессии — это зона Tessera (отзыв). Совпадение учётки с живой сессией определяется по
числовому UID ИЛИ по имени учётки.

#### Scenario: Удаление учётки с живой сессией
- **WHEN** план содержит Delete для роль-учётки, у которой в реестре есть активная сессия
- **THEN** `userdel` не выполняется, учётка помечается отложенной, остальной план применяется

#### Scenario: Удаление учётки без живой сессии
- **WHEN** план содержит Delete для роль-учётки, которой нет в реестре живых сессий
- **THEN** удаление выполняется штатно

### Requirement: Удержание владения отложенной учёткой

При отсрочке удаления apply ДОЛЖЕН (MUST) сохранить отложенную учётку в реестре managed с её
прежним `from_version`. Последующий `apply` после завершения сессии ДОЛЖЕН (MUST) снова увидеть
её как managed и довести удаление. Census НЕ ДОЛЖЕН (MUST NOT) терять владение отложенной
учёткой (иначе она стала бы чужой и более не удалялась бы).

#### Scenario: Повторный apply после закрытия сессии
- **WHEN** учётка была отложена, сессия закрылась, реестр живых сессий её больше не содержит
- **THEN** повторный apply удаляет учётку, реестр managed её больше не содержит

### Requirement: Группы отложенной учётки не сносятся

При отсрочке удаления учётки apply НЕ ДОЛЖЕН (MUST NOT) удалять группы, в которых эта учётка
состоит (первичную или дополнительные). Любое `groupdel` для группы, принадлежащей отложенной
учётке, ДОЛЖНО (MUST) быть исключено из исполняемого плана, а такая группа ДОЛЖНА (MUST)
удерживаться в реестре managed вместе с учёткой. Иначе живая сессия лишилась бы группы (разрыв
доступа), а `groupdel` первичной группы активного uid провалился бы и откатил весь apply.

#### Scenario: Удаление учётки с живой сессией и её роль-группы
- **WHEN** план удаляет роль-учётку с живой сессией и её роль-группу, где учётка — единственный член
- **THEN** удаление учётки и группы откладывается, группа удерживается в реестре, живая сессия цела

### Requirement: Стандалон без реестра сессий

Если файл реестра живых сессий отсутствует (`NotFound`), apply ДОЛЖЕН (MUST) трактовать набор
живых сессий как пустой и применять деструктив штатно. Механизм §12 относится к Tessera-managed
сессиям; без Tessera роль-учётки не gated этим слоем, и Census не должен блокироваться.

#### Scenario: Файл реестра отсутствует
- **WHEN** реестр живых сессий `/run/tessera/sessions.json` не существует
- **THEN** apply считает живых сессий нет и выполняет удаления штатно

### Requirement: Fail-closed на нечитаемом реестре

apply ДОЛЖЕН (MUST) прерваться до снапшота и любой мутации с ненулевым кодом (fail-closed),
если файл реестра живых сессий присутствует, но не читается или содержит битый JSON, И план
содержит хотя бы один Delete: недоказуемое отсутствие сессий не должно вести к обрыву живой
сессии. План без Delete НЕ ДОЛЖЕН (MUST NOT) блокироваться ошибкой чтения реестра.

#### Scenario: Битый реестр и деструктивный план
- **WHEN** реестр существует, но JSON битый, и план содержит Delete
- **THEN** apply отказывает до мутаций, ОС не изменена

#### Scenario: Битый реестр и неразрушающий план
- **WHEN** реестр существует, но JSON битый, и план не содержит ни одного Delete
- **THEN** ошибка чтения реестра игнорируется, план применяется

### Requirement: Развязка от типов Tessera

Census ДОЛЖЕН (MUST) читать реестр живых сессий лениво — извлекая только поля `pam_user` (имя
роль-учётки) и `uid`, игнорируя остальные. Census НЕ ДОЛЖЕН (MUST NOT) зависеть от Rust-типов
Tessera для этого чтения; контракт — два поля JSON, а не общий тип. Эволюция схемы реестра
Tessera (новые поля) НЕ ДОЛЖНА (MUST NOT) ломать чтение Census.

#### Scenario: Реестр с дополнительными полями
- **WHEN** запись сессии содержит поля помимо `pam_user`/`uid` (cert CN, host id, и пр.)
- **THEN** Census успешно извлекает имя и uid, прочие поля игнорирует

### Requirement: Сигнализация отложенных удалений

`census apply` ДОЛЖЕН (MUST) сообщать каждую отложенную учётку (имя и uid) и завершаться
ненулевым кодом, отличимым от ошибки фазы (частичный успех, требуется повтор), при ≥1 отсрочке.
Применённая часть плана и обновление реестра managed (с удержанием отложенных) ДОЛЖНЫ (MUST)
быть зафиксированы до завершения.

#### Scenario: Сводка с отложенными
- **WHEN** apply отложил M удалений и применил остальной план
- **THEN** печатается сводка (применено N, отложено M), процесс завершается ненулевым кодом, реестр обновлён
