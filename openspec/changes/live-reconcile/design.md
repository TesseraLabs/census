# Design: live-reconcile

## Контракт чтения реестра Tessera

- **Путь** (дефолт): `/run/tessera/sessions.json`. Конфигурируем флагом `--sessions-file P`
  (тесты/нестандартный runtime-каталог).
- **Формат**: JSON-массив объектов. Census десериализует **лениво** — структура
  `SessionEntry { pam_user: String, uid: u32 }`; serde по умолчанию игнорирует неизвестные
  ключи, поэтому **новые** поля в схеме Tessera Census не ломают. Оба нужных поля
  **обязательны** (без `#[serde(default)]`): запись без `pam_user`/`uid` или с неверным
  типом — это повреждение реестра, а не «пустая сессия». Парсинг строгий: один битый элемент
  валит разбор всего массива → трактуется как нечитаемый реестр (fail-режим ниже). Это
  безопасное направление: молчаливо «обнулить» набор живых сессий из-за битой записи нельзя —
  иначе `userdel` оборвал бы живую сессию. Не импортируем `tessera_*` крейты — Census и
  Tessera развязаны (контракт = два поля JSON, не Rust-тип).
- **Чтение root-only `0600`**: Census и так работает root → читает.
- **Атомарность Tessera-писателя**: monitord публикует файл через temp + rename, поэтому
  Census всегда видит целостный снапшот (нет частичного JSON). Битый/обрезанный JSON →
  трактуется как ошибка чтения (см. fail-режим ниже), не как «пустой реестр».
- **Файл отсутствует** (`NotFound`) → пустой набор живых сессий (стандалон, §12).
- **Файл есть, но не читается/битый JSON** → это НЕ «нет сессий». Безопасный режим:
  apply прерывается до фазы удаления с ошибкой (fail-closed) — мы не можем доказать,
  что сессий нет, а молчаливое продолжение оборвало бы живую сессию. Не-деструктивный
  план (без `Delete`) при этом не блокируется (нечего откладывать → реестр не нужен).

## Seam: `SessionSource`

```text
trait SessionSource {
    /// Множество роль-учёток с живой сессией (по имени И по uid).
    fn live(&self) -> Result<LiveSessions, SessionError>;
}
struct LiveSessions { names: HashSet<String>, uids: HashSet<u32> }
```

- `LiveSessionSource { path: PathBuf }` — продакшн: читает файл, лениво парсит, маппит
  `pam_user`/`uid` в множества. `NotFound` → пустой `LiveSessions` (Ok). Прочий IO/parse
  error → `Err`.
- `FakeSessionSource { live: LiveSessions }` — юнит-тесты оркестратора без файловой системы
  (тот же паттерн, что `FakeProvisioner` / `FakeInspector`).
- Матч учётки = совпадение **по uid ИЛИ по имени** (uid стабилен, имя — вторичный ключ;
  достаточно любого).

## Размещение в оркестраторе apply

Порядок (`apply::run`) с врезкой live-reconcile:

```text
1. trust (fail-closed)
2-3. resolve targets
4-5. diff (accounts) + group diff
5c. [НОВОЕ] live-reconcile:
      live = session_source.live()?         // fail-closed на битом реестре, если есть Delete
      для каждого Action::Delete{name} в плане:
        если name/uid ∈ live → пометить deferred, ИЗЪЯТЬ из исполняемого плана
6. anti-lockout gate (на УЖЕ урезанном плане — deferral только снижает риск)
7. snapshot + фазы (create→update→delete[без deferred])
   build_managed_set: целевой набор ∪ deferred-учётки (с их прежним from_version)
```

- **Изъятие до gate** корректно: отложенная учётка остаётся → путей входа не меньше, gate
  не может из-за этого ложно сработать (а наоборот — безопаснее).
- **Удержание в реестре** — критично. `build_managed_set` сейчас строит из `targets`
  (целевой набор декларации); deleted-учёток там нет. Для каждой deferred-учётки добавляем
  запись из `managed_now` (прежний `ManagedAccount` с исходным `from_version`) → следующий
  apply снова увидит её как managed и доудалит, когда сессия закроется. Без этого Census
  «забыл бы» владение → учётка стала бы чужой (foreign), и Census её больше не тронул бы.
- **deferred ≠ mutation**: пропущенный `userdel` не инкрементит счётчик мутаций. Если после
  изъятия план пуст (был только deferred-delete) — идемпотентная ветка, но `registry_written`
  ДОЛЖЕН учесть удержание deferred (реестр уже их содержит → перезапись не нужна, но и не
  должна их потерять).

## Вывод и exit-code

- Каждая отложенная учётка → строка лога:
  `deferred delete of <name> (uid <uid>): live session present`.
- `apply` с ≥1 deferred завершается ненулевым кодом (отличимым от ошибки фазы — это «частичный
  успех, требуется повтор», не сбой). Применённая часть зафиксирована, реестр обновлён
  (с удержанием deferred). CLI печатает сводку: применено N, отложено M.
- `--sessions-file` и дефолтный путь логируются в trust/решение-строки для прозрачности.

## Альтернативы (отклонены)

- **IPC к monitord** вместо чтения файла — лишний контракт/демон-зависимость; файл уже
  атомарный и root-only, читать проще и без рантайм-связи (Census может бежать, когда
  monitord стоит). §17 закрыт в пользу файла.
- **force-delete живой сессии** — Census не убивает сессии (зона Tessera); `userdel` над
  активным uid частично провалится и оставит мусор. Defer безопаснее.
- **logind/utmp fallback при отсутствии файла** — новая ОС-вариативная зависимость
  (Astra vs Debian), §12 явно про Tessera-managed сессии; стандалон без Tessera не gated
  этим слоем. Отклонено в пользу простоты.

## Тестирование

- Unit (FakeSessionSource): delete с живой сессией → изъят, помечен deferred, удержан в
  managed с прежним from_version; delete без сессии → выполнен; пустой реестр → все deletes
  выполнены; non-delete план игнорирует реестр; deferral перед gate не ломает anti-lockout.
- Unit (LiveSessionSource): парсинг реального формата `ActiveSession` (лишние поля игнор);
  NotFound → пустой Ok; битый JSON → Err; матч по uid и по имени.
- Контейнер: реальный sessions.json с активной учёткой → apply откладывает userdel, учётка
  жива, реестр держит её, exit nonzero; повтор без файла → доудаление.
