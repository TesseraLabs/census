# Proposal: provisioning-doctor

## Why

`census plan`/`apply` управляют состоянием, но у оператора нет команды **проверить**, что
устройство в безопасном инварианте: роль-учётки недостижимы помимо Tessera (§8), реестр
managed сходится с фактом (§4), остался путь входа (§7). Сейчас это проверяется только
косвенно. Нужны read-only диагностики `census doctor` (проверки + ненулевой код при проблеме,
для мониторинга/CI) и `census status` (что под управлением, версия, drift).

Канон: `tessera-ws/specs/2026-06-18-census-core-spec.md` §4, §7, §8, §14.

## What Changes

- Новая capability **provisioning-doctor**: команды `census doctor` и `census status`,
  обе **read-only** (ни одной мутации).
- **`doctor`** проверяет:
  - **Недостижимость (§8)**: каждая managed роль-учётка — пароль заблокирован (shadow `!`/`*`),
    нет `~/.ssh/authorized_keys`, учётка существует. (Полный анализ PAM-стека «ни один сервис
    не пускает помимо tessera» — advisory/вне ядра, см. design.)
  - **Целостность маркера (§4)**: каждая запись реестра managed соответствует факту
    (uid/shell/группы); учётка с GECOS-меткой Census, но **без записи в реестре** →
    флаг (возможно чужая/поддельная).
  - **Anti-lockout (§7)**: остался хотя бы один путь входа вне managed (rescue) — advisory.
  - **Drift** (если задана декларация): непустой план = расхождение (отчёт).
- **`status`**: список managed-учёток + `from_version` каждой; персист-версия декларации;
  сводка drift (если декларация задана).
- **Коды возврата**: `doctor` — ненулевой при нарушении инварианта недостижимости (§8) или
  целостности реестра (§4); drift и anti-lockout — предупреждения (не валят). `status` — 0.

## Non-goals (отдельно)

- Полный анализ PAM-графа (какие сервисы реально пускают в учётку) — advisory-эвристика
  здесь; глубокий разбор pam.d — возможное будущее.
- Мутации/автопочинка (doctor только сообщает; чинит `apply`).
- Агрегированный doctor по парку (census-enterprise).
- Проверка parsec-метки-потолка (зона МКЦ/ParsecBackend, не Census).
