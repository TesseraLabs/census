# Proposal: group-provisioning

## Why

`census apply` назначает членство роль-учётки в группах через `useradd -G`/`usermod -G`, но
**не создаёт сами группы** — они должны существовать на устройстве заранее (иначе `useradd -G`
падает). Это footgun развёртывания: оператор обязан вручную `groupadd` все группы из
`payload.groups` ролей до `apply` (задокументировано как ограничение в
`docs/authoring-packages.md` §12). Census должен сам приводить группы в соответствие
декларации — как он делает с учётками.

Канон: `tessera-ws/specs/2026-06-18-census-core-spec.md` §3 («их членство в группах … из
состава роли») и §5 («создать группы → …») — описывают создание групп; реализация его не
делала. Этот change закрывает gap.

## What Changes

- Новая capability **group-provisioning**: `census apply` создаёт отсутствующие группы,
  на которые ссылаются роли, и управляет ими как managed-объектами.
- **Создание**: группа из `payload.groups` любой роль-учётки, которой нет в системе →
  `groupadd`. Census заносит её в реестр managed (как учётки/sudoers).
- **Стабильный GID (опц.)**: декларация может пинить GID группы (`[[group]] name/gid`) для
  стабильности по парку (audit/NFS); без пина — GID назначает ОС. Конфликт пина (GID занят
  чужой группой) → отказ, не перезапись.
- **Удаление**: managed-группа, на которую больше не ссылается ни одна роль и которой нет в
  декларации, удаляется (`groupdel`) — только если **создана Census** (есть в реестре);
  пред-существующие/чужие группы **не трогаются** никогда.
- **Порядок фаз**: создание групп — **до** создания/обновления учёток (членство требует
  существующей группы); удаление групп — **после** удаления учёток (нельзя снять группу, пока
  есть члены). Включается в существующий full-file backup (`/etc/group`,`/etc/gshadow` уже в
  снапшоте) и rollback.
- **doctor**: managed-группа пропала / GID разошёлся с реестром → находка (целостность §4).

## Non-goals (отдельно)

- Перенумерация GID существующих чужих групп — не делаем (Census владеет только своими).
- Вложенные/системные группы, GID-диапазоны политики — вне ядра.
- Создание групп для `sudo_role`, не входящих в `payload.groups` — sudo через свой
  `sudoers.d`-файл, отдельная группа под него не требуется.
