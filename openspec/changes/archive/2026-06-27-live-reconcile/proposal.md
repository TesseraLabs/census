# Change: live-reconcile

## Why

Канон-спека §12 (`census-core-spec.md`) требует: деструктив над роль-учёткой с **живой**
сессией не должен рвать эту сессию. Tessera-monitord уже ведёт реестр живых сессий —
`/run/tessera/sessions.json` (root-only `0600`, атомарная замена, JSON-массив `ActiveSession`
с полями `pam_user` = имя роль-учётки и `uid`). Сегодня `census apply` этого реестра не читает:
план с `Delete` над роль-учёткой, в которой прямо сейчас работает инженер, выполнит `userdel`
и оборвёт живую сессию (а над активным uid `userdel` ещё и частично провалится). Закрываем
открытый вопрос §17 («протокол: sessions.json vs IPC») — решение: **чтение файла**.

## What Changes

- Census перед фазой удаления читает реестр живых сессий Tessera и **откладывает** `userdel`
  для каждой роль-учётки с активной сессией (политика **defer + warn**).
- Отложенная учётка **остаётся в реестре managed** с прежним `from_version` — Census не теряет
  владение; следующий `apply` после завершения сессии доудалит её.
- Не-деструктивные операции (членство групп, лимиты, sudo) применяются как обычно — живая
  сессия не рвётся, ОС-семантика подхватит изменения на новых сессиях (§12).
- Census **никогда** не убивает сессии — это зона Tessera (отзыв). Census только пропускает
  свой деструктив.
- Стандалон-инвариант: реестр отсутствует (нет Tessera / не запущена) → пустой набор живых
  сессий, деструктив применяется (§12 — про Tessera-managed сессии; без Tessera роль-учётки
  не gated этим механизмом).
- Реестр читается лениво (только `pam_user` + `uid`, прочие поля игнорируются) — нет связки
  Census с типами Tessera; эволюция схемы Tessera Census не ломает.
- `apply` сообщает отложенные удаления и завершается **ненулевым кодом** (CI/мониторинг видит
  незавершённый деструктив), при этом применённая часть плана зафиксирована.

## Impact

- Affected specs: новая capability `live-reconcile`.
- Affected code (Rust, `census`): `apply.rs` (порядок: чтение сессий → фильтр deferred-delete
  перед фазой удаления; deferred-учётки удерживаются в `ApplyReport.managed`), новый seam
  `SessionSource` (trait) + `LiveSessionSource` (читает `/run/tessera/sessions.json`) +
  `FakeSessionSource` (юнит-тесты), `model.rs`/`state.rs` (удержание deferred в managed-наборе),
  `cli.rs` (wiring пути реестра, exit-code), вывод `apply`.
- Без изменений: trust, anti-lockout (deferral только снижает риск lockout — учётка остаётся),
  doctor/status/plan, формат декларации.
