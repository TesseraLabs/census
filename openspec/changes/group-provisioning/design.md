# Design: group-provisioning

## Контекст

`apply` материализует роль-учётки, но членство (`useradd -G <list>`) требует существующих
групп. Этот change добавляет управление группами в тот же декларативный sync: создать
недостающие, удалить осиротевшие (только созданные Census), опц. пиннить GID.

## Решения

### Р1. Источник «требуемых групп»

Требуемый набор групп = **объединение `payload.groups`** по всем роль-учёткам декларации
(прочитано из role-store, как состав) **∪** имена в декларативном блоке `[[group]]`. Группа,
не упомянутая ни одной ролью и не объявленная — не требуется.

### Р2. Опциональный пин GID — расширение декларации

```toml
[[group]]
name = "atm-operators"
gid  = 8010              # опц.; для стабильности по парку (audit/NFS)
```
`Declaration` получает `groups: Vec<GroupSpec>` (`#[serde(default)]`), `GroupSpec { name,
gid: Option<u32> }`. Для требуемой группы: если объявлена с `gid` → `groupadd -g <gid>`; иначе
`groupadd` (GID назначает ОС). Конфликт пина (GID занят другой группой) → отказ.

### Р3. Реестр managed расширяется группами

`managed.toml` получает секцию managed-групп: `name`, `gid`, `from_version`. Census **создаёт/
удаляет/трогает только группы из реестра**. Сценарии diff требуемое vs реестр+факт:
- требуемая группа **отсутствует** в системе → `groupadd` (+ пин GID) → занести в реестр;
- требуемая группа **существует, но НЕ в реестре** (пред-существующая/чужая) → **не трогать**,
  не заносить (Census её не создавал — не владеет). Членство всё равно назначится.
- группа **в реестре**, но больше не требуется и не в декларации → `groupdel` (осиротевшая,
  создана Census);
- группа в реестре, но её GID разошёлся с пином декларации → находка/отказ (не перенумеровываем
  на лету — это деструктив для владельцев файлов).

### Р4. Порядок фаз apply

```
backup (group/gshadow уже в снапшоте — §6 full-file)
  1. создать недостающие группы           ← ДО учёток (членство требует группы)
  2. создать/обновить учётки + членство
  3. sudoers
  4. удалить исчезнувшие учётки
  5. удалить осиротевшие managed-группы    ← ПОСЛЕ учёток (нельзя удалить группу с членами)
реестр (учётки + группы) — атомарно последним
```
При сбое любой фазы — full-file rollback (group/gshadow восстановятся из снапшота).

### Р5. Provisioner / diff

- `Provisioner` += `create_group(name, gid: Option<u32>)`, `delete_group(name)`.
- diff += группо-действия: `GroupAction::{Create{name,gid}, Delete{name}}`. Вычисляются из
  требуемого набора (Р1) vs managed-групп реестра + живого факта (через `SystemInspector`/getent
  group — узнать, существует ли группа и её GID).
- `план` (`census plan`) показывает группо-действия наряду с учётками.

## Безопасность

- Census **никогда** не трогает группу вне реестра managed — закрывает риск удаления/
  перенумерации чужой группы (как маркер учёток §4, авторитет — реестр).
- GID-пин конфликт → отказ (не перезапись), как UID-конфликт §10.
- `/etc/group`,`/etc/gshadow` уже в full-file backup → group-фазы откатываются атомарно.
- Угроза: осиротевшая managed-группа с чужими членами — `groupdel` ОС откажет, если группа
  первичная для кого-то; для доп-групп удаление безопасно. Census удаляет только свои.

## Тестирование

- **Unit**: вычисление требуемого набора (union payload.groups ∪ [[group]]); diff (создать/
  пропустить-существующую-чужую/удалить-осиротевшую); GID-пин в argv; конфликт GID → отказ;
  порядок фаз (группы до учёток, groupdel после userdel) через FakeProvisioner.
- **Контейнер** (harness): декларация с группой без пред-создания → `apply` создаёт группу +
  учётку с членством; роль теряет группу → осиротевшая managed-группа удаляется; пред-
  существующая чужая группа (создана не Census) не удаляется; GID-пин даёт заданный GID.

## Открытые вопросы

- Первичная группа роль-учётки: сейчас `useradd -m` создаёт user-private group по умолчанию —
  оставляем дефолт ОС (не управляем первичной GID в этом срезе); пин касается доп-групп.
- groupdel при наличии членов из других (не-managed) источников — полагаемся на отказ ОС;
  doctor может предупреждать.
